Chapter 01

Complexity Analysis & Big O Notation

Imagine you're looking for a friend's name in a phone book. You could start at page 1 and check every single name — that works, but it's slow. Or you could open the book to the middle, see if your friend's name comes before or after that page, and keep halving. Both approaches work, but one is dramatically faster as the book gets bigger.

Big O notation is how we describe that difference precisely. It tells us how an algorithm's running time grows as the input size n increases — not the exact number of seconds, but the shape of the growth curve.

Key insight: Big O doesn't measure speed in milliseconds. It measures scalability — how does the work change when we double the input? That's what matters when you go from 1,000 users to 1,000,000.

The Big O Family

NotationNameIntuition
O(1)ConstantSame work no matter the input size. Like checking if a light switch is on.
O(log n)LogarithmicHalves the problem each step. Like binary search in a phone book.
O(n)LinearTouch each element once. Like counting people in a line.
O(n log n)LinearithmicEfficient sorting (Merge Sort, Arrays.sort). Slightly more than linear.
O(n²)QuadraticEvery element compared with every other. Like a round-robin tournament.
O(n³)CubicTriple nested loops. Matrix multiplication (naïve).
O(2ⁿ)ExponentialDoubles with each added element. Like listing all subsets.
O(n!)FactorialAll permutations. Practical only for tiny inputs (n ≤ 12 or so).

How to Calculate Big O: A Step-by-Step Method

Here's a reliable process you can follow for any piece of code:

  1. Identify each operation — assignments, comparisons, arithmetic are all O(1).
  2. Count loop iterations — how many times does each loop run relative to n?
  3. Multiply nested loops — if an inner loop runs m times for each iteration of an outer loop that runs n times, that's O(n × m).
  4. Add sequential blocks — if two loops run one after the other, add them: O(n) + O(n) = O(n).
  5. Drop constants and lower-order terms — O(3n + 5) becomes O(n); O(n² + n) becomes O(n²).

Java Examples with Complexity Walkthroughs

Example 1 — O(1): Constant Time

Java
// O(1) — Constant Time
// No matter how big the array is, we always do the same work.
public static int getFirst(int[] arr) {
    return arr[0];  // One operation, always
}

Walkthrough: There's exactly one array access regardless of whether arr has 10 elements or 10 million. No loops, no recursion — the work is fixed. → O(1)

Example 2 — O(n): Linear Time

Java
// O(n) — Linear Time
// We visit every element exactly once.
public static int sum(int[] arr) {
    int total = 0;              // O(1)
    for (int i = 0; i < arr.length; i++) {  // Runs n times
        total += arr[i];         // O(1) per iteration
    }
    return total;               // O(1)
}

Walkthrough: The loop executes n times (once for each element). Inside the loop, we do O(1) work. Total: 1 + n × 1 + 1 = n + 2. Drop constants → O(n).

Example 3 — O(n²): Quadratic Time

Java
// O(n²) — Quadratic Time (Bubble Sort)
// Each element potentially compared with every other element.
public static void bubbleSort(int[] arr) {
    int n = arr.length;
    for (int i = 0; i < n - 1; i++) {         // Outer: runs n-1 times
        for (int j = 0; j < n - i - 1; j++) { // Inner: runs n-i-1 times
            if (arr[j] > arr[j + 1]) {       // Compare adjacent
                int temp = arr[j];              // Swap if needed
                arr[j] = arr[j + 1];
                arr[j + 1] = temp;
            }
        }
    }
}

Walkthrough: The inner loop runs (n-1) + (n-2) + … + 1 = n(n-1)/2 times total. That's n²/2 - n/2. Drop the constant ½ and the lower-order term → O(n²).

Example 4 — O(log n): Logarithmic Time

Java
// O(log n) — Logarithmic Time
// Each iteration cuts the problem in half.
public static int binarySearch(int[] arr, int target) {
    int low = 0, high = arr.length - 1;
    while (low <= high) {               // How many times can we halve n?
        int mid = low + (high - low) / 2; // Avoids integer overflow
        if (arr[mid] == target) return mid;
        else if (arr[mid] < target) low = mid + 1;
        else high = mid - 1;
    }
    return -1;
}

Walkthrough: Each iteration halves the search space. Starting from n elements: n → n/2 → n/4 → … → 1. That's log₂(n) halvings. → O(log n).

Example 5 — O(n log n): Linearithmic Time

Java
// O(n log n) — Linearithmic Time
// Outer loop: log n (halving step). Inner loop: up to n iterations.
public static void nLogNDemo(int n) {
    for (int i = 1; i < n; i *= 2) {  // i doubles each time → log n iterations
        for (int j = 0; j < n; j++) { // Always runs n times
            // O(1) work
        }
    }
}

Walkthrough: The outer loop runs log₂(n) times (i doubles: 1, 2, 4, 8, … n). For each outer iteration, the inner loop runs exactly n times. Multiply: log(n) × n = O(n log n).

Practice Problems

+
Q1: What is the time complexity of this code?
for (int i = 0; i < n; i++) { for (int j = 0; j < 100; j++) { /* O(1) */ } }

Answer: O(n)

The outer loop runs n times. The inner loop always runs exactly 100 times — that's a constant, not dependent on n. So the total work is 100 × n = 100n. Drop the constant → O(n).

Common trap: students see nested loops and immediately think O(n²). But the inner loop bound must depend on n for that to be the case.

+
Q2: What is the time complexity?
for (int i = 0; i < n; i++) { /* O(1) */ }   for (int j = 0; j < n; j++) { for (int k = 0; k < n; k++) { /* O(1) */ } }

Answer: O(n²)

The first loop contributes O(n). The second block has nested loops: n × n = O(n²). Sequential blocks are added: O(n) + O(n²) = O(n² + n). Drop the lower-order term → O(n²).

+
Q3: What is the time complexity?
for (int i = n; i >= 1; i /= 2) { for (int j = 0; j < i; j++) { /* O(1) */ } }

Answer: O(n)

The outer loop halves i each time: n, n/2, n/4, …, 1. The inner loop runs i times. Total iterations: n + n/2 + n/4 + … + 1 = 2n - 1 (geometric series). Drop constant → O(n).

This is a classic trick question — the nested structure suggests O(n log n) or worse, but the geometric sum collapses to linear.

Chapter 02

Recursion

Recursion is when a function calls itself to solve a smaller version of the same problem. Think of it like Russian nesting dolls — you open one, and inside is a slightly smaller version of the same doll, until you reach the tiniest one at the center (the base case).

Every recursive function needs two things:

  1. Base case — the simplest scenario where the answer is immediate (stops the recursion).
  2. Recursive case — break the problem into a smaller sub-problem and call yourself.
Warning: Forgetting the base case → infinite recursion → StackOverflowError. Always define your stopping condition first.

Example 1 — Factorial (Linear Recursion)

Java
// Factorial: n! = n × (n-1) × ... × 1
// Time: O(n) — makes n recursive calls
// Space: O(n) — n frames on the call stack
public static int factorial(int n) {
    if (n <= 1) return 1;      // Base case: 0! = 1! = 1
    return n * factorial(n - 1); // Recursive case: n × (n-1)!
}
// factorial(4) → 4 * factorial(3)
//              → 4 * 3 * factorial(2)
//              → 4 * 3 * 2 * factorial(1)
//              → 4 * 3 * 2 * 1 = 24

Complexity: Each call reduces n by 1, so there are n calls. Each call does O(1) work (one multiplication). Time = O(n). Space = O(n) (call stack depth).

Example 2 — Fibonacci (Tree Recursion)

Java
// Fibonacci: fib(n) = fib(n-1) + fib(n-2)
// Time: O(2ⁿ) — each call branches into two
// Space: O(n) — max depth of the call tree
public static int fib(int n) {
    if (n <= 1) return n;             // Base: fib(0)=0, fib(1)=1
    return fib(n - 1) + fib(n - 2);  // Two recursive calls!
}
// This is VERY slow — fib(40) takes billions of operations.
// The fix? Memoization (cache results) → O(n).

Complexity: Each call makes two recursive calls, creating a binary tree of calls. The tree has depth n, so roughly 2ⁿ nodes. Time = O(2ⁿ). Space = O(n) (max stack depth).

Example 3 — Power Function (Divide & Conquer)

Java
// Efficient power: x^n using divide & conquer
// Time: O(log n) — halves the exponent each step
// Space: O(log n) — recursion depth
public static double power(double x, int n) {
    if (n == 0) return 1;          // Base: x⁰ = 1
    double half = power(x, n / 2); // Solve half the problem
    if (n % 2 == 0) {
        return half * half;          // Even: x^n = (x^(n/2))²
    } else {
        return half * half * x;      // Odd: x^n = (x^(n/2))² × x
    }
}

Complexity: We halve n at each level. Depth = log₂(n). Each level does O(1) work. Time = O(log n). Space = O(log n).

Example 4 — Sum of Array (Linear Recursion)

Java
// Recursively sum array elements
// Time: O(n) — visits each element once
// Space: O(n) — recursion depth equals array length
public static int sumArray(int[] arr, int index) {
    if (index >= arr.length) return 0;   // Base: past the end
    return arr[index] + sumArray(arr, index + 1);  // Add current + rest
}
// Usage: sumArray(new int[]{3, 1, 4, 1, 5}, 0) → 14

Example 5 — Check if Palindrome (Tail-style Recursion)

Java
// Check if a string is a palindrome (reads same forwards & backwards)
// Time: O(n) — at most n/2 comparisons
// Space: O(n) — recursion depth n/2
public static boolean isPalindrome(String s, int left, int right) {
    if (left >= right) return true;      // Base: pointers met
    if (s.charAt(left) != s.charAt(right))
        return false;                     // Mismatch found
    return isPalindrome(s, left + 1, right - 1); // Check inner chars
}
// Usage: isPalindrome("racecar", 0, 6) → true

Practice Problems

+
Q1: Trace the execution of factorial(5). What value is returned?

Trace:

factorial(5) → 5 × factorial(4) → 5 × 4 × factorial(3) → 5 × 4 × 3 × factorial(2) → 5 × 4 × 3 × 2 × factorial(1) → 5 × 4 × 3 × 2 × 1 = 120

5 recursive calls are made before hitting the base case (n=1). The stack unwinds, multiplying as it returns.

+
Q2: How many times is fib() called when computing fib(6)?

Answer: 25 calls

fib(6) calls fib(5) and fib(4). fib(5) calls fib(4) and fib(3)… and so on. The call tree has 25 nodes total. Notice fib(3) is computed 4 separate times! This is why naïve Fibonacci is O(2ⁿ) — massive redundant work.

+
Q3: What's the time complexity of this recursive function?
void mystery(int n) { if (n <= 0) return; mystery(n / 3); mystery(n / 3); }

Answer: O(nlog₃2) ≈ O(n0.63)

Each call makes 2 sub-calls of size n/3. Using the Master Theorem: T(n) = 2T(n/3) + O(1). Here a=2, b=3, and since log₃(2) ≈ 0.63 > 0 (the case where recursion dominates), we get T(n) = O(nlog₃2). This is sub-linear — faster than O(n)!

Chapter 03

Linear Search vs Binary Search

Searching is one of the most fundamental operations in computing. Let's compare two approaches:

Linear search is like looking for a book by checking every shelf from left to right. Binary search is like looking for a word in a dictionary — you open to the middle, decide if your word is in the first or second half, and keep halving.

Critical trade-off: Binary search is dramatically faster (O(log n) vs O(n)), but it requires a sorted array. Linear search works on any array, sorted or not.

Side-by-Side Implementation

Linear Search — O(n)

Java
// Linear Search: check every element from start to end
// Time: O(n) — worst case: element is last or not present
// Space: O(1) — no extra memory needed
// Works on: ANY array (sorted or unsorted)
public static int linearSearch(int[] arr, int target) {
    for (int i = 0; i < arr.length; i++) {
        if (arr[i] == target) {
            return i;   // Found it! Return index
        }
    }
    return -1;           // Not found
}

Binary Search — O(log n)

Java
// Binary Search: halve the search space each step
// Time: O(log n) — halves n each iteration
// Space: O(1) — iterative version uses constant memory
// REQUIRES: sorted array
public static int binarySearch(int[] arr, int target) {
    int low = 0, high = arr.length - 1;
    while (low <= high) {
        int mid = low + (high - low) / 2;  // Safe midpoint
        if (arr[mid] == target) {
            return mid;           // Found!
        } else if (arr[mid] < target) {
            low = mid + 1;          // Target in right half
        } else {
            high = mid - 1;         // Target in left half
        }
    }
    return -1;                    // Not found
}

Performance Comparison

To find an element in a sorted array of 1,000,000 elements:

  • Linear search: up to 1,000,000 comparisons (worst case)
  • Binary search: at most 20 comparisons (log₂ 1,000,000 ≈ 20)

That's a 50,000× difference. And it only grows as data gets larger.

Practice Problems

+
Q1: You have an unsorted array of 10,000 elements and need to search for one value. Should you sort it first and use binary search, or just use linear search?

Answer: Just use linear search.

Sorting takes O(n log n) ≈ 130,000 operations. Then binary search takes O(log n) ≈ 13. Total ≈ 130,013. Linear search takes at most 10,000 operations. For a single search, linear is faster. The sort-then-binary approach only pays off when you search many times.

+
Q2: Binary search on [2, 5, 8, 12, 16, 23, 38, 56, 72, 91] for target 23. Trace each step.

Trace:

Step 1: low=0, high=9, mid=4 → arr[4]=16 → 16 < 23 → low = 5

Step 2: low=5, high=9, mid=7 → arr[7]=56 → 56 > 23 → high = 6

Step 3: low=5, high=6, mid=5 → arr[5]=23 → Found at index 5!

Only 3 comparisons for a 10-element array. Linear search would have taken 6.

Chapter 04

Arrays: Insertion, Deletion & Binary Search

Arrays are the most fundamental data structure — a contiguous block of memory where elements sit side by side. This contiguity is what gives arrays O(1) random access, but it also means insertion and deletion require shifting elements, which costs O(n).

Insertion at Index

To insert at position i, every element from index i onward must shift right by one to make room.

Java
// Insert element at a given index in an array
// Time: O(n) — worst case: insert at index 0, shift all n elements
// Space: O(1) — shifts in-place (assumes array has capacity)
public static int insertAt(int[] arr, int size, int index, int value) {
    // Shift elements right to make room
    for (int i = size - 1; i >= index; i--) {
        arr[i + 1] = arr[i];   // Move each element one position right
    }
    arr[index] = value;       // Place the new element
    return size + 1;           // New size
}
// Example: arr = [10, 20, 30, 40, _, _], size = 4
// insertAt(arr, 4, 1, 15)  → [10, 15, 20, 30, 40, _], size = 5

Complexity: Inserting at index 0 shifts all n elements → O(n). Inserting at the end shifts zero → O(1). Average = O(n/2) = O(n).

Deletion at Index

To delete at position i, every element after i shifts left by one to fill the gap.

Java
// Delete element at a given index
// Time: O(n) — worst case: delete at index 0, shift all elements
// Space: O(1) — in-place operation
public static int deleteAt(int[] arr, int size, int index) {
    // Shift elements left to fill the gap
    for (int i = index; i < size - 1; i++) {
        arr[i] = arr[i + 1];   // Move each element one position left
    }
    return size - 1;            // New size
}
// Example: arr = [10, 15, 20, 30, 40], size = 5
// deleteAt(arr, 5, 1)  → [10, 20, 30, 40, _], size = 4

Complexity: Same logic as insertion. Deleting from index 0 shifts all → O(n). Deleting the last element shifts nothing → O(1). → O(n) in general.

Binary Search on Arrays

As we saw in Chapter 3, binary search on a sorted array gives us O(log n) lookup. Here's the full implementation with detailed comments:

Java
// Binary Search — complete implementation for arrays
// Prerequisites: array MUST be sorted in ascending order
// Time: O(log n)  |  Space: O(1)
public static int binarySearchArray(int[] arr, int size, int target) {
    int low = 0;
    int high = size - 1;

    while (low <= high) {
        // Calculate mid safely (avoids overflow for large arrays)
        int mid = low + (high - low) / 2;

        if (arr[mid] == target) {
            return mid;                // Target found at mid
        } else if (arr[mid] < target) {
            low = mid + 1;              // Discard left half
        } else {
            high = mid - 1;             // Discard right half
        }
    }
    return -1;  // Target not in array
}

Practice Problems

+
Q1: You have arr = [5, 10, 15, 20, 25, 30]. You call insertAt(arr, 6, 2, 12). What does the array look like after? How many shifts occurred?

Result: [5, 10, 12, 15, 20, 25, 30]

Shifts: Elements at indices 5, 4, 3, 2 all shift right → 4 shifts.

Elements 30→idx6, 25→idx5, 20→idx4, 15→idx3. Then 12 goes into index 2.

+
Q2: Why is insert at the beginning O(n) but insert at the end O(1)?

Inserting at index 0 means every existing element must shift right by one position to make room — that's n shifts for n elements. Inserting at the end (index = size) requires zero shifts — you just place the element in the next open slot. The shifting is what makes array insertion expensive, and the amount of shifting depends on where you insert.

+
Q3: Trace a binary search for target = 8 on arr = [1, 3, 5, 7, 8, 12, 15, 19]. How many comparisons?

Step 1: low=0, high=7, mid=3 → arr[3]=7 → 7 < 8 → low = 4

Step 2: low=4, high=7, mid=5 → arr[5]=12 → 12 > 8 → high = 4

Step 3: low=4, high=4, mid=4 → arr[4]=8 → Found at index 4!

3 comparisons for an 8-element array. log₂(8) = 3, which matches perfectly.
